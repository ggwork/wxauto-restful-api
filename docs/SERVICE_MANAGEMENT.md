# 服务管理机制说明

## 📋 概述

项目实现了智能服务管理机制，可以自动处理端口冲突和避免重复启动。

## ✨ 功能特性

### 1. 智能端口分配
- ✅ 自动检测端口是否被占用
- ✅ 如果首选端口被占用，自动递增寻找可用端口
- ✅ 最多尝试 100 个端口
- ✅ 启动时显示实际使用的端口

### 2. 重复启动检测
- ✅ 启动前检查是否已有运行中的服务
- ✅ 验证服务进程是否存活
- ✅ 验证服务是否健康（可访问）
- ✅ 如果服务正常运行，提示用户而不重复启动

### 3. 服务状态管理
- ✅ 启动后保存服务状态到 `data/service_status.json`
- ✅ 记录端口号、进程ID、启动时间等信息
- ✅ 服务停止时自动清理状态

### 4. 启动锁机制
- ✅ 使用文件锁防止并发启动
- ✅ 锁文件超时自动释放（5分钟）
- ✅ 确保同一时间只有一个启动进程

## 🔄 工作流程

### 启动流程

```
1. 运行 python run.py
   ↓
2. 检查是否有运行中的服务
   ↓
3a. 如果有运行中的服务
   → 验证进程是否存活
   → 验证服务是否健康
   → 显示服务信息并退出
   ↓
3b. 如果没有运行中的服务
   → 尝试获取启动锁
   → 检查首选端口是否可用
   → 如果被占用，递增端口直到找到可用端口
   → 启动服务
   → 保存服务状态
```

### 状态检查

服务管理器通过以下方式验证服务状态：

1. **进程检查**: 使用 `psutil` 检查进程是否存在
2. **健康检查**: 访问 `/docs` 端点验证服务可访问性
3. **状态清理**: 如果服务不健康，自动清理状态文件

## 📁 相关文件

### 核心文件

```
app/services/service_manager.py  # 服务管理器实现
run.py                           # 集成服务管理的启动脚本
~/.wxautox/service_status.json  # 服务状态记录（自动生成）
~/.wxautox/service.lock         # 启动锁文件（自动生成）
```

### 状态文件位置

**重要**: 状态文件保存在用户主目录 `~/.wxautox/`，而不是项目目录。

**原因**:
- ✅ 避免项目相对路径的问题
- ✅ 即使从不同目录或项目副本启动，也能检测到已运行的服务
- ✅ 真正实现全局的服务状态管理

**路径示例**:
- Windows: `C:\Users\<用户名>\.wxautox\service_status.json`
- Linux/Mac: `/home/<用户名>/.wxautox/service_status.json`

### 状态文件格式

`data/service_status.json`:
```json
{
  "service_name": "wxauto-restful-api",
  "port": 8000,
  "pid": 12345,
  "start_time": "2026-03-09T15:30:00",
  "status": "running",
  "host": "0.0.0.0",
  "reload": true,
  "docs_url": "http://127.0.0.1:8000/docs"
}
```

## 🛠️ 实现细节

### 服务管理器 API

```python
from app.services.service_manager import get_service_manager

# 获取服务管理器实例
manager = get_service_manager()

# 启动服务
service_info = manager.start_service(
    preferred_port=8000,
    max_attempts=100,
    force_restart=False
)

# 获取服务信息
info = manager.get_service_info()

# 停止服务
manager.stop_service()
```

### 端口检测原理

```python
def check_port_available(port: int) -> bool:
    """尝试绑定端口来检测是否可用"""
    try:
        with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
            s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
            s.bind(('127.0.0.1', port))
            return True
    except OSError:
        return False
```

### 健康检查原理

```python
def is_service_healthy(port: int) -> bool:
    """访问/docs端点验证服务健康"""
    try:
        response = requests.get(
            f"http://127.0.0.1:{port}/docs",
            timeout=2
        )
        return response.status_code == 200
    except:
        return False
```

## ⚠️ 注意事项

### 1. 端口范围
- 默认从配置文件中的端口开始
- 最多尝试 100 个端口
- 如果 100 个端口都被占用，会抛出异常

### 2. 状态文件
- 状态文件保存在 `data/service_status.json`
- 不建议手动修改此文件
- 服务异常退出时可能需要手动清理

### 3. 进程检测
- 使用进程ID验证服务是否运行
- 如果进程被杀死但状态文件未清理，下次启动会自动清理

### 4. 并发启动
- 使用文件锁防止并发启动
- 锁文件超时时间为 5 分钟
- 如果锁文件存在且未超时，会拒绝启动

## 🔍 故障排查

### 问题1: 无法找到可用端口

**现象**:
```
RuntimeError: 无法在 8000-8100 范围内找到可用端口
```

**解决方案**:
1. 检查是否有大量程序占用端口
2. 修改配置文件中的首选端口
3. 增加 `max_attempts` 参数

### 问题2: 显示服务已运行但实际没有

**原因**: 状态文件未清理

**解决方案**:
```bash
# 手动删除状态文件（在用户主目录）
# Windows
del %USERPROFILE%\.wxautox\service_status.json

# Linux/Mac
rm ~/.wxautox/service_status.json

# 重新启动服务
python run.py
```

### 问题3: 锁文件导致无法启动

**现象**:
```
无法获取启动锁，可能有其他进程正在启动服务
```

**解决方案**:
```bash
# 如果确认没有其他进程在启动，删除锁文件（在用户主目录）
# Windows
del %USERPROFILE%\.wxautox\service.lock

# Linux/Mac
rm ~/.wxautox/service.lock

# 或等待5分钟让锁自动超时
```
